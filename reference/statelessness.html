Reference · Statelessness

C++20/23 services on containers.

Twelve reference documents (~42,000 words) on C++20/23 service design for Linux containers, focused on statelessness as a deployment property — what makes a service safe to kill, replace, and replicate at the orchestrator's discretion, and how the C++ language features, library choices, and operational patterns either support that property or quietly undermine it.

Companion to the main tutorial; cross-references back into §3 (RAII), §7 (Memory), §9 (Networking), and §11 (Noisy neighbors) where the topics overlap.

Reading order

Sequential reading is the default — each document builds on the prior. If time is limited, the irreducible minimum is 01 (vocabulary), 04 (the State Architecture Table), and 10 (the integration). These three cover the architectural shape without the depth on individual mechanisms.

Cross-cutting themes

Eleven themes recur across the documents; they're worth keeping in mind across topics.

  1. Orchestrator state ≠ language state — statelessness is a deployment configuration choice
  2. Podman primary, Kubernetes noted where it materially differs
  3. Request-scope vs process-scope is the practical seam, enforced by RAII
  4. Threading is the third axis of statelessness (TLS, coroutines, gRPC threading model)
  5. RAII has a performance side, not just a correctness side
  6. std:: containers need explicit choices in a service context
  7. OS container requests and limits shape C++ behaviour in non-obvious ways
  8. "Container" is overloaded vocabulary — std:: containers, OS containers, Kubernetes containers
  9. C++ pays a "globals tax" that 12-factor languages don't
  10. gRPC C++ is the through-line for the example service
  11. OTel + Grafana provides external observability that makes statelessness work

Stack assumptions

  • Language: C++20 portable baseline; C++23 features called out where they help
  • Compiler: GCC 14+ (preferred) or Clang 18+ with libc++
  • Build: Conan 2.x, CMake 3.27+, Ninja
  • Runtime: Linux with cgroups v2
  • Services: gRPC C++ (callback API), OpenTelemetry C++ SDK
  • Backing services: PostgreSQL (libpqxx), Redis (redis-plus-plus), Kafka (librdkafka)
  • Orchestration: Podman + podman-compose (primary), Kubernetes (noted where it materially differs)
  • Observability: Grafana stack — Loki for logs, Tempo for traces, Mimir for metrics

Reference books

The patterns and recommendations draw from a small number of well-thought-out sources. Each document's annotated bibliography calls out the chapters most relevant to that topic. Consolidated list:

  • Iglberger, C++ Software Design (Addison-Wesley, 2022)
  • Yonts, 100 C++ Mistakes and How to Avoid Them (Manning, 2024)
  • Enberg, Latency: Reduce Delay in Software Systems (Pragmatic, 2024)
  • Andrist & Sehr, C++ High Performance, 2nd edition (Packt, 2020)
  • Ghosh, Building Low Latency Applications with C++ (Packt, 2023)
  • Geewax, API Design Patterns (Manning, 2021)
  • Vernon, Implementing Domain-Driven Design (Addison-Wesley, 2013) — for the Outbox pattern
  • The Twelve-Factor App (Heroku, 2011/2017) — 12factor.net

Provenance

This reference set was developed as a companion to the Optimizing C++ on Containers tutorial; it grew out of a sidebar request during demo-06 development on the topic of statelessness — a topic the four main reference books treat obliquely (each from its own angle) but no single one integrates with the OS-container deployment realities. The research notes preserve the working drafts and per-document framing decisions.

Documents use an _rNN revision suffix in their source filenames (omitted from URLs here); the initial baseline is _r01. Subsequent revisions bump the suffix with prior revisions retained in git history.
